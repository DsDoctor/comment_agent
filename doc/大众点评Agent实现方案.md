# 大众点评 Agent 实现方案

## 1. 项目目标

本项目希望把一次线下真实体验，整理成可以直接发布到大众点评的图文内容。

用户只需要提供：

- 店铺或地点名称
- 城市、商圈或大致地址
- 体验类型：餐厅、酒店、打卡地等
- 几张重要实拍照片
- 简单体验感受，例如口味、环境、服务、排队、价格、推荐点、避雷点

Agent 负责完成：

- 查询店铺基础信息和公开评价参考
- 优化实拍照片，并添加手绘风格标记
- 生成 100-200 字范围内、可直接发布的大众点评文案
- 将原始素材、处理结果、文案和信息来源保存到本地归档

## 2. 核心工作流

### 2.1 用户输入

一次点评任务建议包含以下信息：

```yaml
place_name: 店名或地点名
city: 城市
area: 商圈或附近地标，可选
category: restaurant | hotel | spot
visit_date: 体验日期
user_notes: 用户自己的真实体验感受
photos:
  - 原始照片 1
  - 原始照片 2
  - 原始照片 3
style_preference: 种草 / 客观 / 俏皮 / 高级感 / 避雷
```

其中 `user_notes` 是最重要的信息源。最终点评应该优先表达用户真实体验，网上信息只作为背景补充，不能伪造用户没有经历过的内容。

### 2.2 信息补全

Agent 可以根据店名和城市检索公开信息，重点补全：

- 地址、营业时间、人均价格、分类标签
- 招牌菜、热门房型、热门拍照点等
- 排名靠前评价中高频出现的关键词
- 常见优点与常见槽点

输出时建议保留信息来源，方便后续复核：

```text
sources.md
- 来源标题
- 来源链接
- 摘要
- 使用到的信息点
```

### 2.3 图片优化

图片处理分为两层。

基础优化：

- 亮度和对比度调整
- 色温校正
- 轻微锐化
- 必要时裁切构图
- 保留原始图片，不覆盖原图

点评增强：

- 添加手绘风格箭头、圈注、下划线
- 添加短标签，例如“这个必点”“景观位”“出片角度”“入口在这”
- 根据图片内容生成不同标记，不做过度装饰

图片输出建议：

```text
output/enhanced_photos/
  01_enhanced.jpg
  02_marked.jpg
  03_cover.jpg
```

### 2.4 文案生成

点评文案要求：

- 100-200 字
- 可直接发布到大众点评
- 带适量表情符号
- 口吻自然，不像广告
- 明确表达真实体验
- 可以包含推荐点，也可以包含轻微避雷提醒

推荐生成多个版本：

- `review_final.md`：最终推荐版本
- `review_options.md`：不同风格备选文案

示例结构：

```text
标题：
正文：
推荐点：
适合人群：
```

最终发布正文可以不强制分段，但本地归档里建议保留结构化版本，方便修改。

## 3. 本地归档结构

每次体验生成一个独立归档目录：

```text
archives/
  2026-05-09_店名/
    input/
      original_photos/
      notes.md
    output/
      enhanced_photos/
      review_final.md
      review_options.md
      metadata.json
      sources.md
```

字段说明：

- `input/original_photos/`：用户原始照片
- `input/notes.md`：用户原始体验感受
- `output/enhanced_photos/`：优化后的图片
- `output/review_final.md`：最终可发布文案
- `output/review_options.md`：备选文案
- `output/metadata.json`：店铺信息、体验日期、分类标签等
- `output/sources.md`：公开信息来源和摘要

## 4. MVP 实现范围

第一版建议先做最小闭环：

1. 用户提供店名、城市、体验感受和照片
2. Agent 查询公开店铺信息
3. Agent 生成店铺信息摘要
4. Agent 生成 100-200 字点评文案
5. Agent 对照片做基础优化和简单手绘标记
6. Agent 保存完整归档

MVP 暂不做复杂平台自动发布，也不直接模拟用户在大众点评网页或 App 内操作。

## 5. 后续扩展

### 5.1 多平台文案

同一份体验可以扩展生成：

- 大众点评版
- 小红书版
- 抖音探店版
- 朋友圈版

不同平台的语气和结构可以分开配置。

### 5.2 评分建议

Agent 可以根据用户感受生成评分建议：

```json
{
  "taste": 4.5,
  "environment": 4.0,
  "service": 4.0,
  "value": 4.2
}
```

评分只作为参考，最终由用户确认。

### 5.3 个人口吻沉淀

后续可以保存用户历史点评，提取个人表达偏好：

- 常用词
- 喜欢的 emoji 密度
- 文案长短
- 推荐和吐槽的平衡方式
- 更偏真实记录还是种草表达

这样生成的点评会越来越像用户本人。

### 5.4 简单本地页面

可以增加一个本地网页工作台：

- 输入店名、城市、体验感受
- 上传照片
- 选择文案风格
- 点击生成
- 预览文案和优化图片
- 一键保存归档

## 6. 技术模块建议

建议拆成以下模块：

```text
comment_agent/
  researcher/      # 查询和整理公开信息
  image_editor/    # 图片优化和标记
  writer/          # 点评文案生成
  archiver/        # 本地归档
  templates/       # 文案和输入模板
  config/          # 配置项
```

每个模块职责保持简单：

- `researcher`：只负责查资料和总结，不写最终文案
- `image_editor`：只负责图片处理，不改用户原图
- `writer`：根据真实体验和资料生成文案
- `archiver`：统一保存输入、输出和来源

## 7. 内容原则

为了让点评可靠、自然、可长期使用，建议遵守：

- 真实体验优先
- 不编造没有体验过的细节
- 网上评价只做参考，不直接照搬
- 保留原图和来源，方便复查
- 文案要像普通用户分享，不要像商业广告
- 避雷内容保持克制，避免无依据的负面判断

## 8. 下一步开发计划

建议按以下顺序开发：

1. 建立项目目录结构和输入模板
2. 实现本地归档逻辑
3. 实现文案生成模板
4. 接入公开信息查询流程
5. 接入图片基础优化流程
6. 增加手绘标记能力
7. 做一个本地页面作为操作入口

第一阶段完成后，就可以用真实店铺素材跑通一次完整点评生成流程。
